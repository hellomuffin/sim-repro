Generative model

A task = scene ├Ś goal predicate, made true by an action chain

A BEHAVIOR activity instantiates a scene (one of 15 interactive houses) populated with the activity's objects in a sampled initial state, plus a goal: a first-order-logic predicate over object states. The robot must execute a chain of action primitives that drives the world from :init to a state satisfying :goal.

Task  =  Scene  ├Ś  Objects  ├Ś  Init state  ├Ś  Goal predicate
solved by a chain of action primitives whose effects are validated by transition rules

Because effects are physically simulated, the goal predicate (e.g. "the mug contains drip-coffee", "the shirt is folded", "the apple is sliced and cooked") can only be satisfied by genuinely producing that state ÔÇö there is no symbolic shortcut.

Task definition language

Anatomy of a BDDL activity

Each of the 1,018 activities is a BDDL problem with three parts. Here is brewing_coffee (abridged):

(define (problem brewing_coffee)
  (:objects
     coffee_maker.n.01_1 - coffee_maker.n.01
     coffee_bean.n.01_1  - coffee_bean.n.01
     water.n.06_1        - water.n.06
     mug.n.04_1          - mug.n.04
     drip_coffee.n.01_1  - drip_coffee.n.01)
  (:init
     (inside coffee_bean.n.01_1 coffee_maker.n.01_1)
     (filled coffee_maker.n.01_1 water.n.06_1)
     (future drip_coffee.n.01_1)         ; product does not exist yet
     (ontop mug.n.04_1 countertop.n.01_1))
  (:goal
     (and (real  drip_coffee.n.01_1)      ; the recipe must actually run
          (contains mug.n.04_1 drip_coffee.n.01_1))))

:objectsTyped entities drawn from the WordNet-style BEHAVIOR object taxonomy
:initInitial predicates (kinematic + extended states); future marks products not yet created
:goalConjunction/quantified predicates to satisfy; real requires a transition rule to have fired
Difficulty

Four complexity axes

A BEHAVIOR episode's hardness is multi-dimensional ÔÇö richer than a simple step count because state changes can be physical/chemical, not just spatial.

1 ┬Ě Action-graph length

Number of primitives in the solving chain (the edges of the action scene graph). graspÔćĺplace = 2; a recipe with gatherÔćĺfillÔćĺtoggleÔćĺwaitÔćĺpour can exceed 10.

2 ┬Ě Room / object complexity

How many rooms the base must traverse and how many distinct objects appear across the trajectory ÔÇö the scene-scale the egocentric policy must parse.

3 ┬Ě State-change depth

How many extended states must change (soaked, sliced, cooked, foldedÔÇŽ) and how many transition rules must fire. Physical changes are irreversible and harder to plan.

4 ┬Ě Manipulation difficulty

Articulation (open fridge/drawer), bimanual handling, fluids & particles, and tool use (knife, sponge) ÔÇö the low-level control burden under the primitive.

On the benchmark page we label each episode with a hardness score = number of edges in its action scene graph, and distinguish easy vs hard by rooms traversed and object complexity ÔÇö the same scheme used for VirtualHome, so the two simulators are directly comparable.
